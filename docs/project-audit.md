# MarkEdit 项目审计报告

- 审计日期：2026-07-11
- 审计范围：`D:\Code\markedit\markdown-editor-v1.html`、`D:\Code\markedit\README.md`
- 审计方式：静态代码审查 + 文档一致性检查
- 审计结论：该项目适合作为**离线原型 / 交互实验**，但当前还不适合继续扩展为面向真实用户的编辑器版本。主要风险集中在：**编辑态切换逻辑、渲染安全、Markdown 保真度、解析/重建不对称**。

## 一、总体评价

### 优点

1. **原型边界明确**：单文件、零构建、零外部网络依赖，便于快速试验。
2. **结构大体清晰**：代码已按解析层 / 渲染层 / 编辑层 / 数据层分段，阅读成本不高。
3. **README 较完整**：已明确记录当前能力、限制和维护约定。
4. **块级编辑思路明确**：`blocks + activeIdx` 的状态模型足够支撑原型迭代。

### 主要结论

- 当前版本最适合“可信内容 + 本地演示”。
- 如果要继续演进，建议优先修复 **P1 编辑切换缺陷** 与 **P1 渲染安全风险**。
- 在这些问题解决前，不建议继续叠加保存、导入、剪贴板、分享等能力，否则风险会被放大。

---

## 二、关键问题清单

| 优先级 | 问题 | 影响 |
| --- | --- | --- |
| P1 | 切换编辑块时使用了重渲染前的旧 DOM 引用 | 可能导致“保存当前块后未正确进入目标块编辑态”，属于核心交互风险 |
| P1 | Markdown 渲染结果直接写入 `innerHTML`，缺少 HTML 清理 | 一旦输入不可信内容，会引入 XSS / 危险链接注入风险 |
| P2 | `parseBlocks()` 对多类文本逐行 `trim()`，破坏 Markdown 原始语义 | 会丢失尾随空格换行、缩进、部分格式意图，导致内容保真度不足 |
| P2 | 解析、渲染、重建三层不对称 | 后续接入保存/导出时容易出现格式损失或能力漂移 |
| P3 | 内置帮助文案与实际快捷键行为不一致 | 用户会被误导，且文档已开始漂移 |
| P3 | 调试输出与未接入主流程代码仍保留在主文件 | 增加噪音，提升维护成本 |

## 三、详细发现

### 1. P1：切换编辑块时存在“旧 DOM 引用”问题

**证据**

- `createBlockEl()` 点击事件中，先执行 `exitEdit(activeIdx)`，再直接对旧 `wrapper` 调用 `enterEdit(wrapper, i)`：
  - `markdown-editor-v1.html:420-424`
- `exitEdit()` 内部会调用 `refreshDisplay()`，而 `refreshDisplay()` 会清空并重建整个 `container`：
  - `markdown-editor-v1.html:481-507`
  - `markdown-editor-v1.html:430-441`

**问题说明**

这意味着：

1. 用户点击目标块 B；
2. 代码先提交当前块 A；
3. `exitEdit()` 触发全量重渲染；
4. 原先点击到的 `wrapper` 已脱离当前文档树；
5. 随后 `enterEdit(wrapper, i)` 仍然对这个旧节点操作。

从静态逻辑看，这里有明显的状态/DOM 生命周期错位风险，属于核心交互缺陷。即使某些浏览器下“看起来偶尔能工作”，也不稳定。

**影响**

- 点击另一个块时，可能只提交当前块，但不能稳定进入目标块编辑态；
- 后续如果继续叠加选区恢复、撤销栈、虚拟列表，这类问题会更难定位；
- 这是编辑器最基础的交互路径之一，优先级应最高。

**建议**

推荐改成以下任一方案：

1. **状态驱动方案（推荐）**  
   不在点击处理器里直接 `enterEdit(wrapper, i)`，而是：
   - 先记录 `nextActiveIdx = i`
   - 提交旧块
   - 在 `refreshDisplay()` 完成后，根据新的索引重新查询 DOM，再进入编辑态

2. **先刷新后查询新节点**  
   在 `exitEdit()` 之后通过 `container.querySelector(...)` 重新拿一次目标块元素，再调用 `enterEdit()`。

3. **避免全量重渲染**  
   长期看，可将“退出旧编辑态 + 进入新编辑态”做成最小 DOM 更新，而不是每次整树重建。

### 2. P1：渲染边界缺少 HTML 清理，存在安全风险

**证据**

- `renderInline()` 直接调用 `marked.parseInline(text)`：
  - `markdown-editor-v1.html:346-352`
- 段落和默认分支直接调用 `marked.parse(block.content)`：
  - `markdown-editor-v1.html:370-400`
- 渲染结果最终通过 `wrapper.innerHTML = renderBlock(block)` 注入页面：
  - `markdown-editor-v1.html:414-418`

**问题说明**

`marked` 是 Markdown 解析器，不是 HTML sanitizer。当前实现允许 Markdown 产出的 HTML 直接进入 DOM。

如果后续支持：

- 打开本地 Markdown 文件
- 粘贴外部内容
- 导入剪贴板 HTML/Markdown
- 多人共享文档

那么恶意 HTML、危险链接协议、事件属性注入等问题都会直接进入渲染面。

**影响**

- 对“可信离线原型”以外的使用方式不安全；
- 一旦接入文件导入、持久化、同步或分享，该问题会立即升级为阻塞项；
- 安全边界目前完全依赖“用户只输入可信内容”的人为约束。

**建议**

- 在渲染边界增加 HTML sanitizer；
- 明确链接协议白名单（例如 `http:`、`https:`、`mailto:`）；
- 如果短期不引入 sanitizer，至少应：
  - 在 UI 和 README 中明确标注“禁止加载不可信内容”；
  - 在后续任何导入/粘贴功能落地前，将该问题列为前置条件。

---

### 3. P2：`parseBlocks()` 逐行 `trim()`，会破坏 Markdown 保真度

**证据**

- 每一行先被 `const trimmed = line.trim();` 处理：
  - `markdown-editor-v1.html:276-277`
- 段落、引用、列表都基于 `trimmed` 写入：
  - `markdown-editor-v1.html:321-338`

**问题说明**

这会带来以下副作用：

1. 段落尾随两个空格产生的 Markdown 硬换行语义会丢失；
2. 某些作者刻意保留的前导/尾随空格会丢失；
3. 引用、列表中的对齐或缩进信息会被抹平；
4. 编辑器在“重新解析后”不能稳定还原用户原始输入。

**影响**

- 当前版本更像“结构化重写器”，而不是“高保真 Markdown 编辑器”；
- 一旦引入保存/导出，用户会遇到“我没改格式，但格式变了”的体验问题；
- 会降低对技术文档、命令示例、手工排版内容的适配能力。

**建议**

- 仅在做“块识别”时使用 `trimmed`，不要把 `trimmed` 直接回写为内容源；
- 保留原始行 `line`，按块类型决定保存哪种规范化结果；
- 为以下输入增加测试：
  - 尾随双空格换行
  - 列表缩进
  - 引用内多行文本
  - 需要保留前导空格的普通文本

---

### 4. P2：解析、渲染、重建不对称，后续扩展容易丢格式

**证据**

- 代码块解析时记录了语言：
  - `markdown-editor-v1.html:286-289`
- 代码块渲染未使用语言信息：
  - `markdown-editor-v1.html:390-391`
- `rebuildDoc()` 重建代码块时也未保留语言：
  - `markdown-editor-v1.html:559-576`
- `renderTable()` 和 `table` 分支存在，但 `parseBlocks()` 中没有生成 `table` 类型的逻辑：
  - `markdown-editor-v1.html:357-363`
  - `markdown-editor-v1.html:393-394`
  - `markdown-editor-v1.html:252-343`
- 列表在解析和渲染时都被压平成简单一层：
  - `markdown-editor-v1.html:329-331`
  - `markdown-editor-v1.html:373-382`

**问题说明**

当前实现中，“能解析什么”“能渲染什么”“能重建什么”并不完全一致。短期看这只是原型限制；长期看则会造成明显的维护漂移。

**影响**

- 以后接入保存、导出、会话恢复时，会出现结构或格式损失；
- 新增块类型时，极易只改动其中一层，导致功能半完成；
- README 已经开始通过说明文字弥补实现差距，说明这种不对称已经在影响认知成本。

**建议**

- 把“支持的块类型”整理成一个明确的能力矩阵：解析 / 渲染 / 重建 / 编辑；
- 未接通主流程的类型（例如 `table`）要么补齐，要么先移除；
- 给 `parseBlocks()` ↔ `rebuildDoc()` 做 round-trip 测试，确保核心语法至少能“输入 -> 编辑 -> 重建”稳定往返。

---

### 5. P3：内置帮助文案与真实快捷键行为不一致

**证据**

- 默认文案写的是“按 Enter（光标在末尾）创建新段落”：
  - `markdown-editor-v1.html:221`
- 实际代码中：
  - 普通 `Enter` 默认插入换行
  - `Ctrl+Enter` / `Cmd+Enter` 才会提交并创建新段落
  - 空块上的 `Enter` 才会直接创建新段落
  - 见 `markdown-editor-v1.html:519-534`

**问题说明**

README 已经描述了正确行为，但页面内置示例文案还停留在旧版本语义。用户第一次打开页面，最先看到的是页面正文，不是 README，因此这里的误导更直接。

**影响**

- 用户会误解快捷键；
- 容易把实际行为误判为 bug；
- 说明维护流程没有完全落实到“代码 + README + 默认示例文案”三处同步。

**建议**

- 立即同步 `DEFAULT_MD` 中的帮助文案；
- 以后把这类文本抽成单独常量，降低更新遗漏概率；
- 最好补一个最小的交互回归检查清单，确保快捷键文案始终和实现一致。

---

### 6. P3：调试输出和未接入主流程代码增加维护噪音

**证据**

- 页面启动时仍输出调试日志：
  - `markdown-editor-v1.html:205-206`
- 以下函数/路径存在但未形成清晰主流程闭环：
  - `renderDoc()`：`markdown-editor-v1.html:410-412`
  - `mergeBlocks()`：`markdown-editor-v1.html:589-593`
  - `renderTable()` / `table` 分支：见上节

**问题说明**

这不属于立即阻塞问题，但会让后续维护者更难判断哪些代码是“当前产品能力”，哪些只是“未来预留”。

**建议**

- 发布版移除无必要 `console.log`；
- 对未接入主流程的函数做一次清理：
  - 要保留，就补注释说明用途与未接入原因；
  - 不保留，就删除，减少歧义。

## 四、工程化观察

### 1. 自动化测试仍然缺失

这在原型期可以接受，但从当前代码形态看，下面这些逻辑已经值得最小单测覆盖：

- `parseBlocks()`
- `exitEdit()` 的块替换 / 分裂逻辑
- `finishBlockAndCreateNext()`
- 快捷键行为（至少做手工用例表）

**建议优先补三类测试数据**：

1. 标题、段落、列表、引用、代码块的基本往返；
2. 一个 textarea 被拆成多个 block 的情况；
3. 含空行、缩进、尾随空格、围栏语言标记的情况。

### 2. 全量重渲染策略适合原型，但不适合继续堆功能

`refreshDisplay()` 当前每次都重建整个容器（`markdown-editor-v1.html:430-441`）。这对 demo 足够简单，但如果继续增加：

- 滚动位置保持
- 光标恢复
- 撤销/重做
- 多选区
- 拖拽排序
- 虚拟化长文档

则会迅速成为瓶颈。

**建议**

- 短期：先保证状态切换正确；
- 中期：将“块内容更新”和“整页重渲染”拆开，逐步转向最小 DOM 更新。

## 五、建议的整改顺序

### 第一阶段：先修阻塞项

1. 修复“切换编辑块时使用旧 DOM 引用”的问题；
2. 明确渲染安全边界，至少先禁止不可信输入场景；
3. 同步默认帮助文案与真实快捷键行为。

### 第二阶段：提升数据可靠性

4. 调整 `parseBlocks()`，减少不必要的 `trim()` 破坏；
5. 为核心块类型建立 round-trip 测试；
6. 明确代码块语言、表格、列表嵌套等能力是否要正式支持。

### 第三阶段：再考虑功能扩展

7. 再接入导入/导出/本地持久化；
8. 逐步减少全量重渲染；
9. 清理死代码，收敛状态模型。

---

## 六、结论

MarkEdit 当前是一个**方向正确、但仍明显处于原型阶段**的项目。它的优点是：目标明确、代码集中、上手成本低；它的问题也很集中：**交互状态切换、渲染安全、格式保真和工程化保障不足**。

如果目标只是继续做原型验证，那么现状可接受；如果目标是继续往“可日常使用的 Markdown 编辑器”推进，建议先完成本报告中的 P1 / P2 项，再继续加功能。
